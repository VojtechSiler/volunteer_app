# AIESEC Opportunities

A prototype web app for browsing AIESEC exchange opportunities, filtering and
searching them, viewing rich detail pages, exploring them on a world map, and —
the headline feature — **matching a candidate's profile to the best-fit
opportunities** with a transparent scoring breakdown.

Built with **React + TypeScript + Vite + Tailwind CSS**. Everything runs in the
browser against a local mock dataset, so there is no backend to set up. It's
structured so the mock data layer and the mock auth layer can be swapped for the
real AIESEC API when it's available.

---

## Prerequisites

This machine does **not** currently have Node.js installed. You need it to run
the app.

1. Install Node.js **20 LTS or newer** from <https://nodejs.org> (the Windows
   installer). Accept the default options.
2. Close and reopen your terminal so `node` and `npm` are on your PATH.
3. Verify:
   ```bash
   node -v
   npm -v
   ```

## Run it

From the `aiesec-app` folder:

```bash
npm install       # first time only
npm run dev       # starts Vite, opens http://localhost:5173
```

Then **register** a new account (any email + a 6+ char password — it's stored
locally in your browser), and you're in.

Other scripts:

```bash
npm run build     # type-check + production build into dist/
npm run preview   # preview the production build
```

> Fonts (Sora, Plus Jakarta Sans) and map tiles (CARTO/OpenStreetMap) load from
> the internet at runtime, so keep a connection for the first load.

### Optional: AI-powered CV auto-fill

The "Paste a CV" button on the matching page works out of the box with an
offline parser. To use OpenAI for higher-accuracy extraction, copy `.env.example`
to `.env` and add your key:

```bash
cp .env.example .env
# then edit .env:  VITE_OPENAI_API_KEY=sk-...
```

Restart `npm run dev` after editing `.env`. Note: Vite inlines the key into the
browser bundle, so this is for **local/prototype use only** — in production, call
OpenAI from a small backend proxy instead of the browser.

## What's inside

| Area | Notes |
|------|-------|
| **Auth** | Register, log in, log out, and change password. Front-end only: accounts live in `localStorage`, passwords are SHA-256 hashed (demo-grade, not production security). See `src/lib/auth.tsx`. |
| **Opportunity list** | Grid / list / map views, full-text search, and faceted filters (programme, duration, region, benefits, field of study, language, SDG focus) plus sorting. `src/pages/Opportunities.tsx`, `src/lib/filters.ts`. |
| **Detail page** | Every field we can surface: description, responsibilities, languages + levels, preferred backgrounds, age range, compensation, what's included (accommodation / meals / transport / visa / pickup), program benefits, deadlines, SDGs, and a location mini-map. `src/pages/OpportunityDetail.tsx`. |
| **Match a person** | Enter a candidate profile → every opportunity is ranked by fit with a "why it matches" breakdown. `src/pages/Match.tsx`, `src/lib/matching.ts`. |
| **Saved candidates** | Save and switch between candidate profiles **within a session** — kept in memory only (never stored to a DB or disk, cleared on refresh/sign-out) to avoid holding candidates' personal data at rest. `src/lib/candidates.tsx`. |
| **CV auto-fill** | Paste a CV or short description to auto-populate the candidate form. Uses OpenAI when a key is set, otherwise an offline heuristic parser. `src/lib/extractProfile.ts`. |
| **Adjustable match weights** | Tune what matters for a given candidate with live sliders; the ranking re-scores instantly. |
| **Map** | Interactive Leaflet world map with programme-coloured pins and popups. `src/components/MapView.tsx`. |
| **Design** | Light/dark themes via CSS-variable tokens (`src/index.css`), Tailwind mapped to those tokens (`tailwind.config.js`). Toggle in the top bar. |
| **Localization** | Full UI in **English / Čeština / Slovenčina**. Switch via the EN/CS/SK control in the top bar (and in Settings); the choice persists and the initial language is detected from the browser. All strings live in `src/lib/i18n.tsx`. |

### Localization notes

Every interface string, form label, filter, and match-reason is translated. The
matching engine (`src/lib/matching.ts`) is i18n-agnostic — it returns translation
*keys* + parameters, which the UI resolves — so explanations localize correctly.
Dates and the salary unit follow the active language.

**Not translated** (by design): opportunity content that would come from the
AIESEC API — titles, descriptions, organisations, city/country names, SDG names —
plus the AIESEC programme brand names (Global Volunteer / Talent / Teacher). To
add a language, extend the `LANGUAGES` array and add its entry to each key in
`DICT` in `src/lib/i18n.tsx`.

### The data model

The mock dataset lives at `src/data/opportunities.json` (160 opportunities) and
is generated by `scripts/generate_mock_data.py`. It reflects the fields from
your crawler's Excel export **plus** the richer fields visible on a real
opportunity page (e.g. `aiesec.org/opportunity/global-talent/1337869`): salary
with currency/period, benefit breakdown, language levels, preferred fields of
study, age range, work schedule, application window, and geo-coordinates for the
map.

Regenerate it any time (requires Python):

```bash
python scripts/generate_mock_data.py
```

The TypeScript shape is defined in `src/types.ts`.

## About the "match a person" feature

The candidate profile captures nationality, age, languages (with proficiency),
field of study, degree level, availability date, preferred duration/programme/
regions, SDG interests, and whether they want a paid role. Each opportunity gets
a **0–100 score** from a weighted, transparent rule engine
(`src/lib/matching.ts`):

| Signal | Weight |
|--------|-------:|
| Languages (meets required level) | 28 |
| Field of study | 18 |
| Availability vs. start window | 14 |
| Age eligibility *(hard limit)* | 12 |
| Duration fit | 10 |
| Interests / SDG overlap | 9 |
| Programme preference | 5 |
| Region preference | 4 |

Age acts as a hard filter (out-of-range roles are ruled out); everything else is
soft-weighted, and each result shows exactly which criteria passed or failed.

### Do you need a vector database / semantic search?

**Not for this.** For a few thousand structured opportunities, a weighted rule
engine is faster, needs zero infrastructure, and is fully explainable — you can
tell a candidate *why* a role ranked first, which a similarity score can't. The
current fields (language, age, dates, study field, SDGs) are categorical or
numeric, so exact/weighted matching is the right tool.

A semantic layer becomes worthwhile only for the **free-text** parts:
- matching a candidate's free-text "what I want to do" against long opportunity
  descriptions,
- fuzzy-matching study fields / job titles that don't share keywords
  (e.g. "growth hacking" ≈ "marketing").

If/when that's needed, the clean integration point is `fieldScore` /
`interestsScore` in `matching.ts`: precompute embeddings for each opportunity
description (e.g. via the AIESEC API + an embeddings model, stored in a vector
DB like pgvector/Pinecone/Qdrant), and blend a cosine-similarity term into the
score alongside the existing rules. The rule engine stays as the explainable
backbone; embeddings just add a "vibe match" component.

## Swapping in the real AIESEC API

Two files are the seams:

- **`src/lib/data.ts`** — replace the JSON import with `fetch` calls to the API.
  Keep the `Opportunity` type (or adapt it) and the rest of the UI is unchanged.
- **`src/lib/auth.tsx`** — replace the `localStorage` implementation of
  `register` / `login` / `changePassword` with real API/token calls. The
  `useAuth()` surface stays identical, so no components need to change.
